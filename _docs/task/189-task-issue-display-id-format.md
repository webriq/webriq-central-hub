# 189: Human-Readable `display_id` for Tasks & Issues

**Created:** 2026-07-24
**Priority:** MEDIUM
**Type:** feature
**Recommended Tier:** deep

---

## Overview

`tasks.id` and `issues.id` are UUIDs with no human-readable identifier. The only
current "display ID" is a raw UUID fragment ‚Äî `TASK ¬∑ {task.id.slice(0, 8).toUpperCase()}`
in `_task-drawer.tsx:100` and `_task-detail.tsx:208` ‚Äî which is neither memorable nor
sequential. This surfaced while discussing task 188 (project route ‚Üí `project_id`);
Zoho's own imported issue records already carry a project-scoped display convention
(`issues.prefix`, e.g. `"TC3-I1"` ‚Äî see migration 051 / task 108), confirming PMs
already recognize this style of ID from Zoho.

**Format, confirmed with the user:**
```
BDD824C501-T0001   (task)
BDD824C501-I0001   (issue)
```
Decomposed:
- `BDD824C5` ‚Äî last 8 hex chars of the owning customer's `customer_id` (same source
  bytes as `projects.project_id`'s own prefix).
- `01` ‚Äî the project's own 2-digit incremental sequence (the same digits that appear
  after `-PROJ-` in that project's `project_id`, e.g. `BDD824C5-PROJ-01` ‚Üí  `01`).
  Concatenated directly onto the customer hex with **no** `-PROJ-` separator, unlike
  `project_id` itself.
- `-T` / `-I` ‚Äî literal type marker (Task / Issue).
- `0001` ‚Äî **4-digit**, zero-padded, incremental **per project, per type** (tasks and
  issues each get their own independent counter starting at `0001` within a project;
  a project's first task is `...-T0001` and its first issue is separately `...-I0001`).
  4 digits (not project_id's 2) because a single project may accumulate thousands of
  tasks over its lifetime.

This mirrors migration 088/task 187's `generate_project_id()` trigger shape almost
exactly (per-scope `pg_advisory_xact_lock`, `max(...)+1` sequence derivation, one-time
backfill of existing rows) ‚Äî reuse that pattern rather than reinventing one.

## Requirements

- [ ] New migration adds a `display_id text unique` column to `tasks` and a separate
      `display_id text unique` column to `issues` (see "Design Decision" below on why
      this is a **new** column rather than reusing `issues.prefix`).
- [ ] `BEFORE INSERT` trigger on `tasks` (mirrors `generate_project_id()`):
  - Resolves the owning project's `project_id`, strips the `-PROJ-` separator
    (`replace(project_id, '-PROJ-', '')`) to get the 10-char base.
  - `pg_advisory_xact_lock` scoped per `(project_id, 'task')` so concurrent inserts for
    the same project don't race on the same sequence number.
  - Sequence = `max(existing numeric suffix for this project's tasks) + 1`, not
    `count(*) + 1` (so a deleted task's number is never reused ‚Äî same rule as
    `project_id`'s own trigger).
  - Zero-padded to 4 digits.
- [ ] Same trigger shape on `issues`, independent counter, `-I` marker.
- [ ] One-time backfill: every existing `tasks` row and every existing `issues` row gets
      a `display_id` generated by the same rule, numbered per project in `created_at`
      order (same approach as migration 088's Part 2b project_id backfill).
- [ ] `tasks_display_id_key` / `issues_display_id_key` unique constraints added after
      backfill (drop-before/add-after pattern only needed if a bulk UPDATE approach
      risks transient collisions ‚Äî direct if unsure, follow migration 088's precedent).
- [ ] `_task-drawer.tsx:100` and `_task-detail.tsx:208` ‚Äî replace
      `task.id.slice(0, 8).toUpperCase()` with `task.display_id`.
- [ ] No issue-display UI exists yet (confirmed: `issues` table is import-only per
      `CLAUDE.md`, no browsing UI shipped) ‚Äî the `issues.display_id` column and trigger
      are added for schema completeness / future-proofing only; no UI change needed for
      issues in this task.
- [ ] `CLAUDE.md`'s `issues` table bullet point gets a short addition noting the new
      `display_id` column and its format, parallel to the existing `prefix` note.

## Design Decision: new `display_id` column, not reusing `issues.prefix`

`issues.prefix` already exists and is populated **only** for Zoho-imported issues, in
Zoho's own format (`"TC3-I1"`, migration 051/task 108). Overwriting it with the new
format for Hub-native issues would leave the column holding two incompatible formats
depending on row origin ‚Äî confusing and inconsistent. Consistent with task 187's
established precedent ("no retroactive rewrite of historical import snapshot data"), a
new `display_id` column is added instead; `prefix` is left untouched for imported rows.
Flag this decision for reconfirmation with the user if it turns out an Issues browsing
UI is being planned imminently and a single unified column is preferred at that time.

## Out of Scope / Must-Not-Change

- **No routing change.** `display_id` is a *display-only* value shown in UI text ‚Äî
  `taskId`/`issueId` route segments (where they exist) keep using the UUID `id`, exactly
  like `project_id` is display-only everywhere except the two documented portfolio-
  tracker/projects exceptions (task 150, task 188). Do not wire `display_id` into any
  URL.
- **No change to `issues.prefix`** ‚Äî legacy imported values stay exactly as imported.
- **No Issues browsing/list UI build-out** ‚Äî that's a separate, not-yet-scoped feature;
  this task only adds the column + trigger + backfill so it's ready when that UI ships.
- **No change to `tasks.id` / `issues.id` (UUID primary keys)** ‚Äî `display_id` is
  additive, not a replacement key. No FK, join, or lookup anywhere switches to
  `display_id`.

## Code Context

### Precedent trigger ‚Äî migration 088's `generate_project_id()`
```sql
create or replace function generate_project_id() returns trigger as $$
declare
  next_seq int;
begin
  if new.project_id is not null then return new; end if;
  perform pg_advisory_xact_lock(hashtext(new.customer_id));
  select coalesce(max(substring(project_id from '-PROJ-(\d+)$')::int), 0) + 1
  into next_seq from projects where customer_id = new.customer_id;
  new.project_id := upper(right(new.customer_id, 8)) || '-PROJ-' || lpad(next_seq::text, 2, '0');
  return new;
end;
$$ language plpgsql;
```
The `tasks`/`issues` triggers follow this exact shape, keyed by `project_id` (the FK
UUID column on those tables) instead of `customer_id`, and deriving the base string from
the parent project's already-generated `project_id` rather than generating fresh hex.
