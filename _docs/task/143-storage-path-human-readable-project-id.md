# 143: Storage Bucket Path — Use Public project_id Instead of Project UUID

**Created:** 2026-07-14
**Priority:** LOW
**Type:** enhancement
**Recommended Tier:** fast
**Status:** Completed

---

## Overview

Follow-up to task 142 (added `projects.project_id`, a public-facing human-readable ID
like `69FF-PROJ-56RT`, auto-generated by a DB trigger — display-only, not a routing
key) and task 137 (nested the Supabase `customer-assets` bucket path under the
project, `{customerId}/{projectId}/{timestamp}_{filename}`).

Today the `{projectId}` segment of that path is the project's raw UUID (`project.id`),
so the bucket looks like:

```
WRQ-CUST-69FF/3f1a9c2e-....－.../1731234567890_brand-guide.pdf
```

Confirmed with the user: swap that segment to the new human-readable `project_id`
instead, so the whole path reads consistently at every level, matching how
`customerId` already reads:

```
WRQ-CUST-69FF/69FF-PROJ-56RT/1731234567890_brand-guide.pdf
```

This is purely a change to *what string gets passed as* `project_id` in the upload
`FormData` — the upload route itself (`src/app/api/customers/[customerId]/assets/upload/route.ts`)
already builds the path from whatever string it's given and needs no change; it has
no opinion on UUID vs. human-readable.

## Requirements

- [ ] `src/app/v2/(hub)/onboarding/[projectId]/page.tsx`'s project query adds
      `project_id` to its `.select(...)` list and passes it through the `project={{ ... }}`
      prop passed to `OnboardingWizard`.
- [ ] `OnboardingWizardProps.project` type (`_onboarding-wizard.tsx`) widens to include
      `project_id: string | null`.
- [ ] All 6 upload call sites in `_onboarding-wizard.tsx` (`formData.append("project_id",
      project.id)` — Documents, Business Facts, Outcome Target, Migration Checklist,
      Content Map, HTML Mockup) change to send `project.project_id ?? project.id` —
      falling back to the UUID only in the theoretical case `project_id` is null (it
      won't be in practice, since migration 066's trigger + backfill populate it for
      every row, but the column is nullable in the type, so the fallback keeps this
      honest rather than asserting non-null).
- [ ] **New uploads only** — following task 137's own established precedent exactly:
      no retroactive move/rename of already-uploaded Storage objects. `file_path` in
      the DB is the source of truth for reads (the `file-url`/`content` routes fetch by
      the stored path, never reconstruct it), so existing rows with a UUID-shaped path
      keep working unchanged; only files uploaded from this point forward get the new
      human-readable path segment.

## Out of Scope / Must-Not-Change

- No change to `src/app/api/customers/[customerId]/assets/upload/route.ts` — it already
  builds `storagePath` from whatever `project_id` string it receives; nothing there
  needs to know or care which ID shape it is.
- No backfill/move of historical Storage objects (same explicit boundary task 137 set).
- No change to `customerId`'s segment — already human-readable, untouched.
- No change to any other `project` field, the onboarding wizard's other logic, or any
  upload call site outside this one file (confirmed via `grep -rn
  "formData.append(\"project_id\""` across `src` — all 6 occurrences are in this file).
- No change to `projects.project_id` generation itself (task 142's trigger) — this task
  only changes what gets read and forwarded, not how the value is produced.

## Code Context

### `page.tsx` — current select + prop (around line 30, 42)

```ts
.select("id, name, customer_id, customers(company_name)")
// ...
project={{ id: project.id, name: project.name, customer_id: project.customer_id, company_name: companyName }}
```

Change to:

```ts
.select("id, name, customer_id, project_id, customers(company_name)")
// ...
project={{ id: project.id, name: project.name, customer_id: project.customer_id, project_id: project.project_id, company_name: companyName }}
```

### `_onboarding-wizard.tsx` — prop type (line 82)

```ts
project: { id: string; name: string; customer_id: string; company_name: string };
```

Change to:

```ts
project: { id: string; name: string; customer_id: string; project_id: string | null; company_name: string };
```

### `_onboarding-wizard.tsx` — the 6 identical upload call sites (lines 580, 625, 687, 869, 941, 1013)

```ts
formData.append("project_id", project.id);
```

Change to (identical replacement at all 6 sites — safe as a single `replace_all` edit,
confirmed byte-for-byte identical at every occurrence):

```ts
formData.append("project_id", project.project_id ?? project.id);
```

## Implementation Notes

### What Changed

  `project.project_id ?? project.id` — a single `replace_all` edit, safe since all 6
  occurrences were confirmed byte-for-byte identical beforehand.

  builds the Storage path from whatever string is passed as `project_id`, agnostic to
  UUID vs. human-readable shape.
